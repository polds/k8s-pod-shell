# Releasing

This project uses a single version source per release: a semver git tag.

## Version contract

- Release tag format: `vX.Y.Z` (example: `v1.2.3`)
- Normalized release version: `X.Y.Z`
- The same normalized version is used for:
  - container image tag (`ghcr.io/<owner>/kubeshell-web:X.Y.Z`)
  - Helm chart version (`version: X.Y.Z`)
  - Helm appVersion (`appVersion: X.Y.Z`)

## Published tags and artifacts

For release `vX.Y.Z`, the workflow publishes:

- Container tags:
  - `ghcr.io/<owner>/kubeshell-web:vX.Y.Z`
  - `ghcr.io/<owner>/kubeshell-web:X.Y.Z`
  - `ghcr.io/<owner>/kubeshell-web:latest`
- Helm chart package:
  - `kubeshell-web-X.Y.Z.tgz`
- Helm chart OCI push:
  - `oci://ghcr.io/<owner>/charts/kubeshell-web:X.Y.Z`

## Release workflow modes

`release.yml` supports:

- **Tag-driven release**: push a git tag `vX.Y.Z`
- **Manual release** (`workflow_dispatch`): provide `version` as `X.Y.Z`

Both modes enforce semver shape and run test/typecheck gates before publish.

Release notes are composed from:

- a custom preamble generated by CI (asset links + install snippets),
- matching version section from `CHANGELOG.md` when present,
- GitHub auto-generated notes (PRs/commits) appended after the custom content.

## Standard release steps

1. Merge release-ready code to `main`.
2. Create and push tag:
   - `git tag vX.Y.Z`
   - `git push origin vX.Y.Z`
3. Wait for `release.yml` to publish image + chart + GitHub Release.
4. Verify:
   - image tags exist in GHCR
   - chart exists in OCI registry
   - GitHub Release includes `kubeshell-web-X.Y.Z.tgz`
